# コントリビューションガイド

MA-Pilotへのコントリビューションを歓迎します。このドキュメントでは、プロジェクトへの貢献方法を説明します。

---

## 行動規範

プロジェクトへの参加者全員に以下を求めます:

- 敬意を持った対話
- 建設的なフィードバック
- 多様性の尊重
- ハラスメントの禁止

---

## コントリビューション方法

### 1. Issue報告

バグや機能提案がある場合、まずIssueを作成してください。

#### バグ報告

以下の情報を含めてください:

- **タイトル**: 簡潔なバグの説明
- **環境**: OS、ブラウザ、バージョン
- **再現手順**: 1. xxx、2. xxx、3. xxx
- **期待される動作**: xxxであるべき
- **実際の動作**: xxxになる
- **スクリーンショット**: あれば添付

#### 機能提案

以下の情報を含めてください:

- **タイトル**: 簡潔な機能の説明
- **背景**: なぜこの機能が必要か
- **提案内容**: 具体的な実装方法
- **代替案**: 他の実装方法（あれば）

### 2. プルリクエスト作成

#### 事前準備

1. Issueを確認し、重複がないか確認
2. 大きな変更の場合、事前にIssueで議論

#### 手順

```bash
# 1. リポジトリをフォーク
# GitHubでForkボタンをクリック

# 2. クローン
git clone https://github.com/YOUR_USERNAME/ma-pilot.git
cd ma-pilot

# 3. ブランチ作成
git checkout -b feature/your-feature-name

# 4. 変更を実施
# コーディング...

# 5. Lint・テスト実行
npm run lint
npm run type-check
pytest

# 6. コミット
git add .
git commit -m "feat: 機能を追加"

# 7. プッシュ
git push origin feature/your-feature-name

# 8. GitHubでプルリクエスト作成
```

#### PRテンプレート

```markdown
## 変更内容

簡潔な変更内容の説明

## 関連Issue

Closes #123

## テスト方法

1. xxx
2. xxx

## チェックリスト

- [ ] Lintパス
- [ ] 型チェックパス
- [ ] テスト追加・実行
- [ ] ドキュメント更新
```

## コーディング規約

### JavaScript/TypeScript

- **ESLint**: `.eslintrc.json`に従う
- **Prettier**: 自動フォーマット使用
- **命名規則**: [開発者ガイド](docs/DEVELOPER_GUIDE.md)参照

### Python

- **PEP 8**: 準拠
- **Ruff**: Linter + Formatter使用
- **型ヒント**: 必須

### コミットメッセージ

Conventional Commits形式:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type**:
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント
- `style`: コードスタイル
- `refactor`: リファクタリング
- `test`: テスト
- `chore`: ビルド・設定

**例**:
```
feat(dashboard): KPI表示機能を追加

売上、患者数、利益率のKPIカードを追加しました。
グラフ表示も実装済みです。

Closes #45
```

## テスト要件

### フロントエンド

- 新機能には必ずテストを追加
- カバレッジ80%以上を維持

### バックエンド

- 全APIエンドポイントに統合テスト
- ビジネスロジックにユニットテスト
- カバレッジ80%以上を維持

## ドキュメント更新要件

以下の変更時はドキュメントも更新してください:

- **API変更**: `docs/API_REFERENCE.md`
- **データベース変更**: `docs/DATABASE_SCHEMA.md`
- **新機能追加**: `docs/USER_MANUAL.md`
- **設定変更**: `README.md`

## レビュープロセス

1. **自動チェック**: CI/CDで自動実行
2. **コードレビュー**: メンテナーがレビュー
3. **修正対応**: 指摘事項に対応
4. **承認**: LGTMで承認
5. **マージ**: mainブランチにマージ

### レビュー基準

- [ ] 機能要件を満たしているか
- [ ] コーディング規約に準拠しているか
- [ ] テストが追加されているか
- [ ] ドキュメントが更新されているか
- [ ] セキュリティリスクがないか
- [ ] パフォーマンス問題がないか
